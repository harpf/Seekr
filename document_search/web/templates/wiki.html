Quick Start

Seekr is a self-hosted, full-text document search engine. It extracts text from PDF, Word, PowerPoint, Markdown, and plain-text files, stores the text in a SQLite FTS5 index, and provides a web UI plus a REST API.

  1. Deploy with docker compose up -d — the app listens on port 8080.
  2. Open the dashboard and sign in (default: admin / admin — change immediately).
  3. Go to Config → Paths, add your document root (e.g. /documents), and click Save.
  4. Go to Ingest → Run Index to crawl and index all supported files.
  5. Open Search and start querying.
The index lives in /data/document_index.db inside the container. Mount a persistent volume at /data to survive restarts.

Search Syntax

The search engine uses SQLite FTS5 with a custom tokeniser. Basic syntax:

SyntaxExampleMeaning
wordinvoiceMatch any block containing "invoice"
word1 word2invoice 2024Implicit AND — both words must appear
"phrase""annual report"Exact phrase match
word*invest*Prefix match (investment, investor…)
-wordinvoice -draftExclude blocks containing "draft"
ORpdf OR docxEither word

Filters

Use the filter panel in the Search UI to narrow results by file type, path prefix, date range, tag, or block type (page, heading, paragraph).

Add additional extensions via Config → General → Supported extensions. Files exceeding max_file_size_mb are skipped.

Tags & Marks

Tags are user-defined labels attached to a document (per user). They are searchable via the tag filter and the tag cloud on the Search page.

Marks (bookmarks) flag a document for quick retrieval. Toggle the bookmark icon on any search result.

Tags set during upload via the API are stored as the uploading user's tags on that document.

Upload & AI Organizer

Files can be uploaded through Ingest → Upload. The optional AI Organizer uses a local Ollama model to suggest a folder sub-path and tags based on the document's content.

Configuring Ollama

  1. Run Ollama and pull a model: ollama pull llama3.2
  2. Set the Ollama URL in Config → AI (default: http://ollama:11434).
  3. Use Test Connection to verify the model responds.

AI Reorganize

Ingest → AI Reorganize scans up to 50 uploaded documents and suggests an optimal folder hierarchy. Accepted suggestions are applied as file moves on disk and path updates in the index.

Home Assistant Integration

Seekr exposes a dedicated REST API for Home Assistant automations, authenticated by long-lived API keys — no session token needed.

API Keys

Go to Config → Home Assistant → Create New Key. Each key is scoped to a path filter — searches via that key only return documents whose path starts with the filter value.

The full key is shown exactly once after creation. Copy it immediately and store it in your HA secrets file.

Use Config → Home Assistant → Connection Test to paste a key and confirm it authenticates successfully before adding it to HA.

HA configuration.yaml snippet

The YAML generator in Config → Home Assistant produces a ready-to-paste snippet. It creates:

  • input_text.seekr_query — a text helper for typing search queries
  • rest_command.seekr_search — calls POST /api/ha/search
  • An automation that fires on query changes and shows results as a persistent notification

Minimal example:

rest_command:
  seekr_search:
    url: "https://seekr.yourdomain.local/api/ha/search"
    method: POST
    headers:
      Content-Type: application/json
      X-Api-Key: !secret seekr_api_key
    payload: '{"query": "{{ query }}", "limit": 5}'

Answer & Sources in the Response

The POST /api/ha/search response includes:

  • answer — a 1–3 sentence plain-text answer generated by the configured Ollama model (or a fallback excerpt if Ollama is unavailable).
  • sources — a list of { filename, path, modified_at } objects for easy attribution.
  • results — full result objects including snippet, extension, and block_type.

Example automation message using the answer:

message: >
  {% if r.content.answer %}{{ r.content.answer }}{% endif %}

  Sources:
  {% for s in r.content.sources %}
  • {{ s.filename }} ({{ s.modified_at }})
  {% endfor %}

API Reference

The full interactive reference is available at /docs (Swagger UI) and /redoc (ReDoc). The sections below highlight the most commonly used endpoints.

Authentication

All UI endpoints require an X-Auth-Token header obtained from POST /api/login. Tokens expire after 8 hours.

Home Assistant endpoints (/api/ha/*) use X-Api-Key instead — no login required.

Administration

Users

Manage users under Config → Users (admin only). Two roles exist: admin (full access) and user (search + mark + tag).

System Updates

Go to Ingest → Check for Updates. If a newer commit is available on GitHub, click Run Update. The app runs git pull && docker compose build --no-cache && docker compose up -d in the background and reconnects automatically.

Database

The SQLite database runs in WAL mode for concurrent reads. Use Config → System → Database Test to check integrity. Use Index Cleanup to remove stale entries for deleted files.

SSL / TLS

Config → SSL lets you generate a self-signed certificate or upload your own. The nginx reverse proxy in docker-compose.yml references /data/ssl/cert.pem and /data/ssl/key.pem.

Editors in the container

Both vim and nano are available inside the container for quick config edits:

docker exec -it seekr-document-search-1 bash
nano /data/config.json

Troubleshooting

SymptomLikely causeFix
Search returns no results Index not built yet Go to Ingest → Run Index
PDF pages missing text Scanned PDF (image-only) Enable OCR — ensure Tesseract is installed (it is in the default image)
HA search returns 401 Wrong or revoked API key Use Config → HA → Connection Test to verify; regenerate key if needed
AI suggestions unavailable Ollama not running or model not pulled Config → AI → Test Connection; pull the model first
Update fails / container won't restart Build error or Docker socket not mounted Check Ingest → Update Status for stderr output; run update manually via SSH
Database locked errors Multiple processes writing simultaneously WAL mode + 5 s busy timeout should handle this; avoid running two indexers concurrently
